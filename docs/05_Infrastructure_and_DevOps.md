このドキュメントは、インフラの構築からアプリケーションのデプロイ、そして本番環境での運用に至るまでの全プロセスを標準化・自動化するための仕様を定義します。

## 1. Infrastructure as Code (IaC) 詳細仕様

インフラはTerraformを用いてコードとして管理します。

### 1.1. Terraform プロジェクト構造

再利用性と可読性を高めるため、モジュールベースの構成を採用します。

```plaintext
terraform/
├── environments/
│   ├── dev/
│   │   └── main.tf       # 開発環境用の変数を定義
│   └── prd/
│       └── main.tf       # 本番環境用の変数を定義
├── modules/
│   ├── vpc/              # ネットワーク関連リソース
│   ├── ecs/              # ECS, Fargate関連リソース
│   ├── rds/              # データベース関連リソース
│   └── iam/              # IAMロール・ポリシー
└── backend.tf            # S3バックエンド設定
```

### 1.2. 状態管理 (State Management)

Terraformの状態ファイル (`.tfstate`)は、チーム開発と状態の安全な保持のため、以下の構成でリモート管理します。

* **バックエンド**: 状態ファイルは**AWS S3**に保存し、チーム全体で単一の信頼できる「公式台帳」を共有します。
* **状態ロック**: 意図しない同時実行による状態ファイルの破損を防ぐため、**AWS DynamoDB**によるステートロッキングを有効にします。

## 2. CI/CDパイプライン詳細設計

CI/CDはGitHub Actionsで実現します。

### 2.1. ブランチ戦略

GitFlowベースの戦略を採用します。

* `main`: **本番環境**にデプロイされている最新のコード。直接のコミットは禁止し、プルリクエスト経由でのマージのみを許可します。
* `develop`: 次期リリースに向けた**開発の統合ブランチ**。これがデフォルトブランチとなります。
* `feature/xxx`: 各機能の開発を行うブランチ。`develop`から分岐し、完了後に`develop`へマージします。

### 2.2. GitHub Actions ワークフロー

**.github/workflows/deploy.yml**

#### `develop`ブランチへのマージ時 (ステージング環境への自動デプロイ)

1.  **Lint & Test**: フロントエンド (ESLint) とバックエンド (`go test ./...`) の静的解析と単体テストを実行。
2.  **Build Docker Image**: バックエンドのGoアプリケーションをコンテナ化し、Dockerイメージをビルドする。タグにはGitのコミットハッシュを使用する。
3.  **Push to ECR**: ビルドしたDockerイメージをAWS ECR (Elastic Container Registry) にプッシュする。
4.  **(Option) DB Migration**: `go-migrate`などのツールを使用し、データベースマイグレーションを実行する。
5.  **Deploy to Staging**: Terraformを実行し、ECSのタスク定義を新しいDockerイメージバージョンに更新し、ECSサービスを再起動してステージング環境にデプロイする。

#### `main`ブランチへのマージ時 (本番環境への自動デプロイ)

* **トリガー**: `develop`ブランチから`main`ブランチへのPull Requestがマージされた時。（※手動承認ステップを挟むことを推奨）
* **パイプラインステージ**: ステージング環境へのデプロイと同様のステップを、本番環境に対して実行する。

## 3. 設定・シークレット管理計画

環境ごとの設定と機密情報を安全に管理します。

1.  **GitHub Secrets (CI/CDパイプラインのためのシークレット)**
    * **役割**: CI/CDパイプライン自体がAWSを操作するために必要な認証情報を保管します。
    * **具体例**: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`
    * **連携**: GitHubリポジトリの設定で安全に保管され、ワークフロー内でのみ環境変数として参照されます。

2.  **AWS Secrets Manager (アプリケーションのためのシークレット)**
    * **役割**: デプロイされたアプリケーション本体が、実行時に必要とする機密情報を保管します。
    * **具体例**: GoバックエンドがRDSに接続するためのデータベースパスワード。
    * **連携**:
        1.  TerraformコードでAWS Secrets Managerにシークレットを作成します。
        2.  ECSのタスク定義で、そのシークレットをコンテナの環境変数として安全に**注入**するよう設定します。
        3.  これにより、アプリケーションはソースコードやコンテナイメージ内にパスワードを一切ハードコーディングすることなく、安全にデータベースへ接続できます。

## 4. E2Eテストケース仕様

リリース前に、アプリケーションの主要な機能が壊れていないことを保証するための最小限の自動テストを定義します。

* **テストケースID**: E2E-HP-001
* **対象ユーザージャーニー**: 新規ユーザー登録から食事を記録し、ダッシュボードに反映されるまでの「ハッピーパス」。
* **使用ツール**: Playwright
* **テスト手順**:
    1.  **Navigate**: アプリケーションのURLにアクセスする。
    2.  **Action**: 「サインアップ」し、新しいユーザーを登録する。
    3.  **Action**: オンボーディング画面でプロフィールと目標を設定する。
    4.  **Action**: ダッシュボード画面で食事記録モーダルを開く。
    5.  **Action**: 「ごはん」を「150g」で記録する。
    6.  **Assert**: ダッシュボードの「摂取カロリー」表示が、ごはん150g分 (252 kcal) 増加していることを**検証**する。
    7.  **Action**: ログアウト操作を行う。
    8.  **Assert**: ログイン画面にリダイレクトされていることを**検証**する。

* **CI/CDへの統合**: このE2EテストはCI/CDパイプラインに組み込み、デプロイ前に必ず実行されるように設定します。
