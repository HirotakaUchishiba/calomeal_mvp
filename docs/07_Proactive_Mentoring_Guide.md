## フェーズ 0: 基盤構築における躓きのポイント

### 1. ローカル開発環境の罠: `.env`ファイル

- **問題点**: `docker-compose up`が`Connection refused`のような汎用的なエラーで失敗する。本来の原因が環境変数の設定ミスであることに気づかず、GoのコードやDockerのネットワーク設定を延々とデバッグしてしまう。
- **原因**: `docker-compose.yml`が参照する`.env`ファイルが存在しない、または設定が不完全なため、バックエンドサービスがデータベースに接続できない。
- **解決策 (ベストプラクティス)**:
    1.  **`.env.example`を作成する**: 必要な環境変数のキーを網羅したテンプレートファイルを作成し、リポジトリでバージョン管理する。
    2.  **`.gitignore`に`.env`を追加する**: 実際のパスワード等が記述された`.env`ファイルがコミットされるのを防ぐ。
    3.  **ローカルで`.env`を作成する**: 各開発者は`.env.example`をコピーして`.env`ファイルを作成し、ローカル環境をセットアップする。

### 2. Terraformの記憶: なぜリモートステートが重要か

- **問題点**: 複数人でTerraformを触ると、他の人が作ったインフラを意図せず破壊したり、インフラの真の状態が誰にも分からなくなったりする。
- **原因**: Terraformが「自分が何を作ったか」を記録している台帳ファイル (`.tfstate`) を個人のPCで管理しているため、各人の台帳に食い違いが生じる。
- **解決策**:
    1.  **リモートステート管理**: `backend.tf`ファイルを設定し、台帳ファイル(`tfstate`)をチーム全員が共有する場所（AWS S3）で一元管理する。
    2.  **ステートロック**: 他の人が作業中にインフラを同時に変更して台帳ファイルを破壊しないよう、排他制御（AWS DynamoDB）を有効にする。

## フェーズ 1-3: 実装における躓きのポイント

### 3. バックエンド最難関: PostgreSQL全文検索の仕組み

- **問題点**: 食品検索機能で、単純な`LIKE '%query%'`検索を実装してしまい、データ増加に伴いパフォーマンスが著しく悪化する。
- **原因**: `CREATE INDEX ... USING GIN (to_tsvector(...))`という高度な機能の仕組みを理解していない。
- **解決策 (概念の理解)**:
    - **`to_tsvector`**: テキストを検索用の「単語リスト」に変換する関数。
    - **`to_tsquery`**: 検索キーワードを検索用の「クエリ形式」に変換する関数。
    - **`@@`演算子**: `tsvector`と`tsquery`を高速にマッチングさせる専用の演算子。
    - **GINインデックス**: `@@`演算子を使った検索を劇的に高速化するための索引。
- **実装**: これらの概念を理解した上で、バックエンドの`searchFood`リゾルバを実装する。

### 4. フロントエンドの悪夢: キャッシュの非同期

- **問題点**: 食事を記録するミューテーションは成功したのに、ダッシュボードの表示が更新されない。手動でページをリロードすると初めて反映される。
- **原因**: サーバー上のデータは更新されたが、フロントエンド（Apollo Client）が持つデータのコピー（キャッシュ）が古いままになっている。
- **解決策**: ミューテーション成功後に、キャッシュを意図的に更新する。
    - **方法1 (シンプル)**: `useMutation`の`refetchQueries`オプションを使い、関連するクエリ (`dailySummary`) を再実行させる。手軽だが、余分な通信が発生する。
    - **方法2 (高パフォーマンス)**: `useMutation`の`update`オプションを使い、キャッシュを直接読み書きして、即座にUIに反映させる。実装は複雑だが、最高のユーザー体験を提供できる。

## フェーズ 4: デプロイにおける躓きのポイント

### 5. 2種類のシークレット管理

- **問題点**: とりあえず全ての機密情報をGitHub Secretsに入れてしまい、セキュリティリスクを高めたり、管理を複雑にしたりする。
- **解決策**: シークレットの種類に応じて保管場所を使い分ける。
    1.  **GitHub Secrets**: **CI/CDパイプライン自体が使う**シークレット（例: AWSへのアクセスキー）を保管する。アプリケーションはこれらの値を知らない。
    2.  **AWS Secrets Manager**: **デプロイされたアプリケーションが実行時に使う**シークレット（例: データベースのパスワード）を保管する。Terraformでこのシークレットを参照し、コンテナの環境変数として安全に注入する。

## 横断的関心事

### 6. AI生成コードへの過信とセキュリティリスク

- **問題点**: AIが生成したコードを無批判に受け入れてしまい、SQLインジェクションなどの深刻なセキュリティ脆弱性を意図せず埋め込んでしまう。
- **原因**: AIはセキュアではないコードを生成する可能性があるという認識の欠如。
- **解決策**:
    1.  **AIの役割再定義**: AIを「熟練プログラマー」ではなく「生産性は高いが間違いを犯しやすいジュニア開発者」と見なす。
    2.  **厳格なレビュー**: 生成されたコードの一行一行を、設計資料とセキュリティのベストプラクティスに照らしてレビューする責任は開発者にある。
    3.  **SASTツールの導入**: `gosec` (Go) や `eslint-plugin-security` (React) といった静的解析セキュリティテストツールをCI/CDパイプラインに統合し、脆弱性を自動的に検出する仕組みを構築する。

### 7. マイクロサービス設計の落とし穴

- **契約破壊**: gRPC/Proto の後方互換性を壊す変更（フィールド削除/再番号付け）
  - 対策: 予約番号の維持、フィールドは非推奨化→段階的削除、バージョニング方針を遵守
- **タイムアウト/リトライ地獄**: デフォルト無期限でハング
  - 対策: クライアントにデッドライン必須、`Unavailable` のみ指数バックオフで限定リトライ
- **識別情報の伝播漏れ**: ユーザー文脈がサービスに届かない
  - 対策: JWT クレームから `x-user-id`/`x-email` をメタデータで必須伝播

### 8. React Native の落とし穴

- **ビルド環境**: Xcode/Android SDK の整合が壊れがち
  - 対策: Expo 管理ワークフローの採用、CI でのキャッシュ戦略
- **認証**: Web の設定を流用して Cognito 設定がズレる
  - 対策: 環境毎の Amplify 設定ファイルを分離し、環境変数で切替
