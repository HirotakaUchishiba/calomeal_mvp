このドキュメントは、栄養管理アプリ「Lean MVP」プロジェクトの基本的な目的、戦略、そして開発範囲を定義します。

## 1.1. プロジェクトビジョンと戦略的目標

本プロジェクトの目的は、ユーザーが日々の食事摂取量や身体活動を記録し、健康目標の達成を支援するWebベースの栄養管理アプリケーションのMVP (Minimum Viable Product) を開発することにあります。

本プロジェクトの核心的な哲学は「Lean MVP」の思想です。最優先目標は、「**ユーザーは栄養を追跡するために食事を記録する**」という中核的なユーザー行動仮説を、最小限のコストと時間で検証することにあります。そのために、「構築-計測-学習」のフィードバックループを最速で回すことを最優先とします。

技術戦略は、GraphQL BFF を境界面とし、バックエンドはドメイン別のマイクロサービスで構成します。マイクロサービス間通信は gRPC（Protocol Buffers 契約駆動）を採用し、クライアントは Web(React) と モバイル(React Native) の二本柱で提供します。Go は標準ライブラリ中心（net/http 等）で運用し、インフラは AWS（ECS Fargate, RDS, ALB, VPC, Cloud Map, CloudWatch 等）を用います。

## 1.2. Lean MVPスコープ定義

当初の仕様はMVPとしては過剰な機能を含んでいたため、ユーザー価値の検証に直結しない、あるいは実装難易度が高い機能を戦略的に延期します。

| 機能分類 | 機能名 | 機能概要 | 当初スコープ | Lean MVPスコープ | 変更の根拠 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **ユーザー管理** | ユーザー登録・認証 | Eメールとパスワードによるアカウント作成とログイン | In | In | 変更なし。基本機能。 |
| | プロフィール・目標設定 | 身長、体重、活動レベル、目標体重・期日などの設定 | In | In | 変更なし。パーソナライズの基本。 |
| **食事記録** | キーワード検索記録 | 食品データベースからキーワードで検索して食事を記録 | In | In | コア機能。ただし、外部APIへの依存をなくし、信頼性の高い内部データベースを利用する仕様に変更。 |
| | バーコードスキャン記録 | 市販品のバーコードをスキャンして栄養情報を自動入力 | In | Out | WebベースSPAでの実装は技術的課題が多く、信頼性も低いため延期。 |

| | 栄養バランスチャート | 栄養素の摂取状況をレーダーチャート等で可視化 | In | Out | 実装コストが高く、コアなフィードバックは日次サマリーで充足できるため延期。 |

### 1.3. プラットフォームスコープ（Web / モバイル）

| プラットフォーム | 範囲 | 備考 |
| :--- | :--- | :--- |
| **Web (React)** | 認証、オンボーディング、ダッシュボード、食事/運動/体重記録 | 既存SPAを継続強化。GraphQL BFF と通信。 |
| **モバイル (React Native)** | 認証、オンボーディング、ダッシュボード、食事/運動/体重記録（最小） | 初期版は Web 同等の最小セット。GraphQL 契約を共有し型生成を共通化。 |

### 1.4. 設計成果物

- GraphQL スキーマ（公開 API 契約）
- gRPC Proto（マイクロサービス間契約）
- E-R 図（当面は共有 RDS、将来はサービス単位に分割）
- アーキテクチャ図（BFF/サービス/インフラ）

### 1.5. 前提・リスクと移行方針

- 当面は RDS を共有し、サービス抽出を優先。将来的にはサービス毎スキーマまたはDB分割へ移行。
- BFF は Cognito JWT を検証し、gRPC メタデータとしてユーザー情報を伝播。
- 契約互換性（GraphQL/Proto）を重視し、後方互換な変更ルールを採用。

### 1.6. 成功指標（例）

- ハッピーパス E2E 成功率（Web/RN）
- 契約互換性維持（GraphQL/Proto）
- 主要ユースケースの p95 レイテンシ（BFF, gRPC）
